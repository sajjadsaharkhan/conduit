# Conduit — Plan & Proposal

## Executive Summary

A **Docker-based proxy client** (sing-box / Xray) that exposes **HTTP, HTTPS, and SOCKS5** proxies inside a container. It includes a **Web UI** for configuration (subscription links, manual configs, domain rules), **admin authentication**, **subscription refresh every 1 minute**, **latency-based node selection**, and **domain-scoped proxying** (only specified domains use the proxy). SQLite is used for persistence.

---

## 1. High-Level Architecture

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          Docker Container                                    │
│  ┌─────────────┐   ┌──────────────┐   ┌─────────────────────────────────┐  │
│  │   Web UI    │   │   Backend    │   │  Proxy Core (Sing-box / Xray)     │  │
│  │  (Frontend) │◄──│  (API + DB)  │──►│  - HTTP proxy  (e.g. 8080)       │  │
│  │  Port 80    │   │  SQLite      │   │  - SOCKS5 proxy (e.g. 1080)      │  │
│  └─────────────┘   │  Scheduler   │   │  - Inbounds + Outbound to node   │  │
│                    │  Latency    │   │  - Domain-based routing           │  │
│                    │  Config Gen │   └─────────────────────────────────┘  │
│                    └──────────────┘                    │                   │
│                             │                          │                   │
│                    Refresh 1min, pick best node         ▼                   │
│                    Generate config, restart core    [Remote V2Ray Server]   │
└─────────────────────────────────────────────────────────────────────────────┘
         │                    │                    │
         ▼                    ▼                    ▼
    Browser (UI)        Local apps           Host machine
                        (optional)           Port mapping:
                                             1080 → SOCKS5
                                             8080 → HTTP
```

- **Web UI**: Served on port 80 inside container; map to host (e.g. `8080:80`) for admin access.
- **Backend**: Handles auth, subscription parsing, config generation, latency checks, and core process lifecycle.
- **Proxy core**: One of Sing-box / Xray; exposes HTTP and SOCKS5 inbounds; outbound = selected node; routing = proxy only for configured domains, direct for the rest.

## 2. Core Technology Choices

| Component        | Recommendation | Rationale |
|-----------------|----------------|-----------|
| **Proxy core**  | **Sing-box** (primary) or **Xray** | Sing-box: single binary, supports VLESS/VMess/Trojan/SS, flexible routing (domain/suffix/keyword/regex), mixed HTTP+SOCKS inbound. Xray: widely used, similar features. |
| **Backend**     | **Python (FastAPI)** or **Node.js (Express/Fastify)** | Fast to implement REST API, subscription parsing (e.g. base64 + vmess/vless parsers), SQLite, subprocess for core. |
| **Frontend**    | **React** or **Vue** (or simple server-rendered HTML) | SPA for settings, subscription URL, domain list, node list + latency, status. |
| **Database**    | **SQLite** | Single file, no extra service, sufficient for one admin, settings, nodes, domains. |
| **Auth**        | Session + hashed password in DB | Admin username/password; store bcrypt hash; session cookie or JWT. |

## 3. Feature Breakdown

### 3.1 Proxy Core Behavior

- **Inbounds** (inside container):
  - HTTP proxy: one port (e.g. **8080**).
  - SOCKS5: one port (e.g. **1080**).
  - Optional: HTTPS (e.g. 8443) if you want CONNECT over TLS; can be added later.
- **Outbound**: One outbound to the currently selected upstream node (VLESS/VMess/Trojan/Shadowsocks, etc.).
- **Routing**:
  - **Domain-based**: Only traffic whose domain matches the configured list goes through the proxy outbound.
  - All other traffic: **direct** (no proxy).
- **Config source**: Generated by the backend from (a) subscription (parsed nodes) or (b) manual JSON / share links; written to a file and core restarted.

### 3.2 Subscription & Config Management

- **Subscription URL**: User enters URL in Web UI; backend stores in SQLite.
- **Refresh interval**: Every **1 minute** a job:
  1. Fetches subscription (HTTP GET).
  2. Decodes (e.g. base64) and parses share links (vmess://, vless://, trojan://, ss://, etc.).
  3. Saves/updates nodes in DB (or in-memory with DB persistence).
  4. Runs **latency check** (e.g. TCP connect or HTTP to a test URL via each node, or use a simple ICMP/tcp ping).
  5. Selects **best node** (e.g. lowest latency).
  6. Generates core config (inbounds + selected outbound + domain rules).
  7. Restarts core process (or sends reload if supported).
- **Manual config**: Optional paste of full JSON config or single share link; backend converts to core format if needed (e.g. share link → outbound) and merges with inbounds + domain rules.

### 3.3 Domain List (Proxy-Only Domains)

- **Web UI**: Editable list of domains (or suffixes/keywords) that must use the proxy.
- **Storage**: SQLite table, e.g. `proxy_domains` (domain/suffix/keyword, type).
- **Config generation**: Core config `route.rules` (Sing-box) or `routing.rules` (Xray):
  - If request domain in list → outbound = proxy.
  - Else → outbound = direct.
- **Default**: Empty list = no traffic proxied; user explicitly adds domains (e.g. `google.com`, `.google.com`, `youtube.com`).

### 3.4 Web UI (Admin Panel)

- **Login**: Username + password; session (or JWT) after successful auth.
- **Pages/sections**:
  - **Dashboard**: Status (core running/stopped), current node, latency, proxy ports.
  - **Subscription**: Subscription URL, last refresh time, “Refresh now” button.
  - **Nodes**: List from subscription (or manual) with latency; select default or “auto (best by latency)”.
  - **Domains**: Add/remove domains (or suffixes/keywords) that should use the proxy.
  - **Manual config**: Paste JSON or share link for non-subscription use.
  - **Settings**: Change admin password, optional: core type (Sing-box/Xray), inbound ports.
- **Tech**: React/Vue + REST API, or server-rendered forms; SQLite-backed auth and settings.

### 3.5 Admin Authentication

- **SQLite**: Table `users` (id, username, password_hash, created_at).
- **Password**: Store bcrypt (or argon2) hash; never plain text.
- **Login**: POST `/api/auth/login` with username/password; set HTTP-only cookie or return JWT.
- **Protection**: All `/api/*` (except login) require valid session/JWT; Web UI redirects to login if 401.

## 4. Data Model (SQLite)

- **users**: id, username, password_hash, created_at
- **settings**: key (e.g. `subscription_url`, `core_type`, `http_port`, `socks_port`), value (JSON or string)
- **proxy_domains**: id, type (domain/suffix/keyword), value, created_at
- **nodes**: id, source (subscription/manual), raw_link, parsed_json, latency_ms, last_check, created_at  
  (or store only “current nodes” in memory and persist subscription URL + last snapshot)
- **optional**: **subscription_snapshots** for history (url, fetched_at, nodes_count)

Simplified: **users**, **settings**, **proxy_domains**, **nodes** (with source and latency).

## 5. Docker Design

- **Base**: Use official Sing-box or Xray Docker image as base, or Alpine/Debian + install binary.
- **Image contents**:
  - Proxy core binary (Sing-box or Xray).
  - Backend (Python/Node) + frontend static files.
  - Entrypoint: start backend; backend starts core when config is ready.
- **Ports** (host:container):
  - `80:80` — Web UI.
  - `8080:8080` — HTTP proxy.
  - `1080:1080` — SOCKS5.
- **Volumes**:
  - Persistent SQLite and config: e.g. `/data` → `./data` (or named volume).
- **Environment**: Optional env for first-time admin username/password or secret for JWT.

## 7. Using as “Proxy for Entire Local Machine”

- **Option A — System proxy**: On the host, set system HTTP/HTTPS proxy to `localhost:8080` and SOCKS5 to `localhost:1080` (or the mapped ports). Only domains that match the configured list will be routed through the container if the core’s routing is domain-based; the rest goes direct from the core.
- **Option B — Per-application**: Point each app (browser, IDE, etc.) to HTTP proxy `localhost:8080` and SOCKS5 `localhost:1080`.
- **Option C — Transparent (advanced)**: Not in initial scope; would require TUN or iptables and host networking considerations.

So: **mapping container ports to host (e.g. 8080, 1080) and setting system or app proxy to those ports** is the supported way to “proxy entire local machine” for the domains you configure in the Web UI.

---

## 8. Security Considerations

- **Admin UI**: Only over HTTP in container; for production, put behind reverse proxy (e.g. Caddy/Nginx) with HTTPS or use TLS for the backend.
- **Default password**: Force password change on first login or set via env.
- **Subscription URL**: Stored in DB; consider not logging it in plain text.
- **Container**: Run as non-root if possible; read-only filesystem where feasible.

---

## 9. Open Points / Alternatives

- **Core**: Support both Sing-box and Xray (switchable in UI) vs. one only (simpler).
- **Latency**: TCP connect vs. HTTP GET to a test URL (e.g. http://www.gstatic.com/generate_204) via each node.
- **HTTPS inbound**: Omit in v1 or add on port 8443 with self-signed cert for CONNECT.
- **Multiple subscriptions**: One URL in v1; later extend to multiple and merge nodes.
